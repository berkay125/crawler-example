For production scale, keep the current architecture boundaries (crawler workers, durable queue state, and searchable index) but swap local components for distributed systems incrementally. A practical path is PostgreSQL for canonical crawl metadata and dedupe state, plus Elasticsearch or OpenSearch for full-text indexing/ranking. Use a message broker such as RabbitMQ or Kafka to decouple URL discovery from fetch workers, which allows horizontal scale-out and resilient retries with dead-letter queues. This preserves the same mental model as the current asyncio queue but with durable, multi-node throughput.

For web-scale reliability and quality, add a distributed scheduler with per-domain rate limits, adaptive backoff, and robots/crawl-delay awareness, then deploy worker pools across regions behind rotating egress IPs/proxies. Add observability (OpenTelemetry traces, Prometheus metrics, structured logs) and a central control plane to tune concurrency, budgets, and blocklist/allowlist policies at runtime. This yields a mature crawler platform that can safely index large corpora while maintaining politeness, fault isolation, and operational visibility.
